# 1. URI란

<aside>
📌

URI는 인터넷상에 있는 특정한 자원(텍스트,이미지,동영상등)을 나타내는 고유한 식별자이다.

- 인터넷 세계에 존재하는 모든 사물에 붙이는 이름표,주소!
</aside>

<aside>
💡

URI는 URL과URN을 아울러서 부르는 용어이다.

- URI = URL + URN

EX) 

URL : “서울시 강남구 00빌딩 2층”(위치)

URN: “홍길동”(장소가 바뀌어도 홍길동은 유지됨)

URI: 이 두 가지 “식별방법”

</aside>

# 1-1. URI이 필요한 이유

<aside>
💡

- **고유한 식별**: 많은 웹페이지,파일 중에서 내가 원하는 데이터를 정확히 찾아내기 위해서 필요하다.
- **통신 규칙**: 서버에 “이 데이터 보내줘”라고 했을때 서버가 이해할 수 있는 공통된 형식의 주소가 필요하다.
- **자원 관리**: 웹 상의 자원을 체계적을 분류하고 접근 권한을 관리하기 위해 표준화된 주소체계가 필요하다.
</aside>

# 1-2. URI는 어디에 쓰이는가?

<aside>
💡

### 1. 웹 브라우저 주소창

“https//www.google.com” 같은것이 URI(URL)이다. 이를 통해 특정 웹서버의 메인 페이지에 접근한다.

### 2. API 설계 (REST API)

서버 데이터를 가져오거나 수정할 떄 URI를 사용한다.

- GET/user/123: “ID가 123인 사용자의 정보를 가져와”
- DELETE/posts/10: “10번 게시물을 삭제해줘”

### 3. HTML문서내부

이미지를 불러오거나 다른페이지로 연결할때 자원의 위치를 URI로 지정한다.

- <img src=”…”>
- <a href=”…”>
</aside>

# 1-3. URI의 구성 요소

<aside>
💡

**https://www.google.com:900/path/to/file?query=abc#fragment**

- Scheme(https): 접근 방법(어떤 규칙으로 대화할지
- Host(www.google.com): 자원이 있는 서버의 주소
- Port(900): 서버 내 접속 관문(생략가능)
- Path(/path/to/file): 서버 내 자원의 상세 경로
- Query(?query=abc): 서버에 전달하는 매개 변수
- Fragment(#fragment): 페이지 내 특정 위치(책갈피)
</aside>

# 2. URL

<aside>
💡

Locator-위치를 이야기한다.

- “어디에 있는가?”가 핵심
- 자원의 위치를 가리킨다.(주소가 바뀌면 해당URL은 유효X)

ex) https://google.com/images/logo.png

</aside>

---

# 3. URN

<aside>
💡

Name-이름을 이야기한다.

- “무엇인가?”가 핵심
- 자원의 이름 그자체(자원의 위치가 바뀌어도 이름은 변하지 않아 영구적이다.)

ex) urn:isbn:9788965402602 (특정 도서의 isbn번호)

</aside>

---

| **구분** | **URL (위치 중심)** | **URN (이름 중심)** |
| --- | --- | --- |
| **의미** | 자원이 **어디**에 있는지 알려줌 | 자원이 **무엇**인지 알려줌 |
| **특징** | 주소가 바뀌면 못 찾음 (가변적) | 위치가 바뀌어도 식별 가능 (영구적) |
| **예시** | `https://naver.com/logo.png` | `urn:isbn:9788960773455` |

# 4. 왜 우리에게 URN보다 URL이 더 익숙할까?

<aside>
💡

### 1. “찾아갈 수 있는가?”의 차이

- **URL:** 주소창에 치면 해당 서버로 찾아가 데이터를 가져온다.
    
    식별과 동시에 위치정보를 주기때문에 바로 결과물 확인 가능
    
- **URN**: 이름만으로 컴퓨터는 “어디로”가야할지 모르기때문에URN은 중간단계(해석기/Resolver)가 필요해서 대중화 되지 않음

### 2. 인터넷(HTTP)프로토콜과 궁합

- 우리가 웹사이트를 볼때 “이름(URN)으로 데이터를 찾아줘”라고 물어 보는게 아니라 “경로(URL)에 있는 데이터를 보내줘”라고 물어본다. 따라서 인터넷 구조 자체가 URL에 최적화 되어있어 익숙하다.

### 3. 중앙 관리의 복잡성

- **URL**: 도메인만 사면 누구나 자유롭게 하위 주소를 만들어 관리 할 수 있음
- **URN**: 전 세계적으로 이름이 중복되지 않게 하는건 까다롭기 때문에 표준을 만들고 관리하는 기구(IETF등)에서 엄격하게 관리해 일반 사용자가 접할 기회가 적음
</aside>
