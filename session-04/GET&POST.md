# GET/POST

<aside>
💡

GET과 POST는 웹 브라우저와 서버가 데이터를 주고받을때 사용하는 가장 대표적인 방식이다.

</aside>

## 1. GET메서드

### 설명

<aside>
💡

- 서버로부터 정보를 조회(Retrieve)할때 사용하는 가장 기본적인 방식이다.
- 웹 브라우저 주소창에 URL을 입력하고 엔터를 누를때 GET요청이 보내진다.
</aside>

### 특징

<aside>
💡

- **데이터 전송 방식(Query String):** 데이터를 URL의 끝에 **?**와 함께 key=value형태로 붙여서 보낸다.

```jsx
ex) http://example.com/find?item=book&color=red
```

- **캐싱(Caching):** 브라우저가 요청 결과를 저장하기 때문에, 동일한 주소로 다시 접속할때 서버에 묻지 않고 빠르게 화면을 보여준다.
- **안전함(Safe):** 서버의 데이터를 변경하거나 삭제하지 않고 ‘읽기’만 수행하기 때문에 안전하다.
- **멱등성(Idempotent):** 같은 요청을 여러번 보내도 서버의 상태와 결과가 동일하다.
    
    ex) 게시글 조회를 여러번해도 목록이 변하지 않는다.
    
- **브라우저 기록(History):**방문 기록에 남는다.
- **길이 제한:** URL길이 제한(약2,000자 내외)여서 대용량 데이터를 보낼땐 적합하지 않다.
</aside>

### 1-3+. 멱등성

<aside>
💡

**연산을 여러번 해도 결과가 변하지 않는 성질**

- **수학적 예시**: f(x) = f(f(x))  ⇒ 숫자 1에 1을 곱해도 계속 1이다.(멱등하다)

**HTTP메서드에서의 멱등성**

- 서버의 상태가 변하는가 기준으로 판단.

| **메서드** | **멱등성** | **설명** |
| --- | --- | --- |
| **GET** | **Yes** | 게시글을 10번 조회해도 서버의 게시글 내용이나 개수는 변하지 않습니다. |
| **PUT** | **Yes** | 정보를 통째로 업데이트합니다. 같은 데이터로 10번 수정해도 최종 결과는 같습니다. |
| **DELETE** | **Yes** | 1번 삭제하나 10번 삭제하나, 해당 데이터가 삭제된 상태라는 결과는 동일합니다. |
| **POST** | **No** | 호출할 때마다 새로운 데이터가 생성됩니다. (결제가 중복되거나 글이 여러 개 써짐)
특징
 |
</aside>

## 1-4. GET의 보안 문제

<aside>
💡

**GET은 데이터가 URL에 노출이 돼어 다음 상황에는 쓰면 안됨**

- **비밀번호나 개인정보**를 전송할 때(주소창,브라우저 기록남음)
- **서버의 상태를 변경**할때 (글 쓰기, 삭제 ,결제등)
</aside>

## 2. POST메서드

<aside>
💡

서버의 리소스를 생성(Create),서버에 데이터 제출(Submit)하여 처리하기 위해 사용

ex) 게시글 작성, 회원가입, 파일업로드

</aside>

## 2-1.특징

<aside>
💡

- **데이터 전송 방식(Message Body):**데이터를 URL이 아닌 HTTP 요청의 Body(본문)에 숨겨서 보낸다.(외부 노출X)
- **데이터 타입 및 용량**: 단순 텍스트뿐만 아니라 JSON, XML, 이미지, 동영상 등 다양한 형식의 대용량 데이터가능
- **캐싱 불가**: 서버의 상태를 변경하는 요청이므로 브라우저가 결과를 캐싱하지 않는다.
</aside>

## 2-2.보안

<aside>
💡

**패킷 분석:** 네트워크를 가로채는 도구를 사용하면 Body에 담긴 내용을 읽을 수 있다.

⇒ **해결:** 반드시 HTTPS를 사용해야 데이터가 안전하다.

</aside>

## 2-3.멱등하지 않다.

<aside>
💡

POST는 호출할 때마다 서버에 새로운 변화가 일어남

ex) “게시글 등록” 4번 → 게시글 5개 생김

⇒ 매번 요청 때미다 결과가 달라짐(비멱등)

</aside>
